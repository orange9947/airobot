# ULN2003 逐相线圈电压诊断设计

## 1. 背景与目标

左右 28BYJ-48 均为 5V 型，四组绕组到红色公共线的电阻均约 24 欧姆。电机 5V 空载约 5.25V，现有低速动作时最低约 4.9V。两块 ULN2003 的指示灯能够按序点亮，零步侧已确认保持关闭，但左右裸轴都只轻微抖动。右侧三种互不等价的四相环形顺序都已测试，结果相同，因此不再继续换相序。

下一步逐一测量右侧 ULN2003 的 A-D 输出与红色公共线之间的带载压差。每个待测通道需要稳定导通 3000ms，让普通实时读数万用表有足够时间显示稳定值，并在任何异常时自动释放线圈。

## 2. 范围

- 首轮只测右侧驱动板的 A、B、C、D，每次只导通一路。
- 左侧驱动板在所有测量中保持关闭。
- 不改正常 `MOVE_WHEELS` 的时序、半步表或完成后释放行为。
- 不使用当前尚未接入输出逻辑的 `hold_ms` 配置冒充诊断保持时间。
- 不向网页、AI 工具或普通机器人控制接口开放逐相诊断。
- 不提交本地照片、临时探针、设备配置或任何凭据。

## 3. 方案选择

采用独立的 `CoilDiagnosticService` 和专用协议命令。与补全全局 `hold_ms` 相比，该方案不会让所有正常运动在结束后继续给线圈通电；与临时测试固件或 OpenOCD 直接写 GPIO 相比，它继续经过正式通信、状态和安全监督链路。

## 4. 模块与接口

### 4.1 STM32 诊断服务

新增与 `MotionService` 分离的线圈诊断服务。服务输入为：

- 目标轮：左或右；
- 目标通道：A、B、C 或 D；
- 导通时间：100-3000ms，本次主机探针固定使用 3000ms；
- 命令 ID 和启动时刻。

启动时服务只向目标轮输出单个通道位，另一轮输出零。到达截止时间后，它先向两轮输出零，再记录完成结果。服务支持显式中止；无论正常完成、中止还是错误，终态都必须是左右输出全零。

线圈诊断与正常运动互斥。任一服务活动时，另一服务的启动请求必须被拒绝。

### 4.2 协议与 ESP32

协议追加线圈诊断请求和结果事件，不改变现有消息布局。请求包含命令 ID、目标轮、目标通道和导通毫秒数；结果事件包含命令 ID 与完成或中止结果。

ESP32 `DeviceService` 只提供内部诊断方法，用于一次性 USB 主机探针。Web API、AI 工具白名单和公开配置不增加相应入口。主机探针每次固定发送一条右轮、指定通道、3000ms 的请求，不自动重试，也不接受连续运行参数。

## 5. 状态与安全门禁

只有同时满足以下条件才允许开始：

- STM32 链路健康；
- 机器人处于 `MANUAL`；
- 不在 ESTOP 或 FAULT；
- 没有正常运动、线圈诊断、资源更新或其他活动命令；
- 目标轮和通道有效，导通时间不超过 3000ms。

`STOP`、ESTOP、FAULT、通信心跳超时、模式切换、资源更新入口和 MCU 复位都必须关闭线圈诊断输出。诊断异常不自动解除 ESTOP，不自动重试，也不自动执行下一通道。

## 6. 单通道测量流程

每个通道独立执行以下流程：

1. 保持电机 5V 断开，在板外用杜邦线分别引出右侧 ULN2003 的 VCC 和待测 A-D 输出，两个末端分开放置。
2. 万用表置直流电压档；红表笔接 VCC 延长线末端，黑表笔接待测输出延长线末端。表笔不直接接触驱动板上相邻引脚。
3. 接通电机 5V，确认两块板指示灯全灭、裸轴安全并可立即物理断电。
4. 在每次命令前明确通知操作员目标通道的指示灯即将亮起，并取得当次确认。
5. 只导通该通道 3000ms，记录稳定的 VCC 到输出压差；到时确认两块板指示灯均熄灭。
6. 断开电机 5V后才能移动杜邦线并准备下一通道。

首轮顺序固定为右侧 A、B、C、D。任一路出现持续异响、明显发热、无法自动熄灭、MCU 异常或读数异常跳变，立即断开 5V并停止后续测试。

## 7. 结果判定

四路先比较相对值，不预设未经实测验证的绝对阈值：

- 单路明显低于其余三路：优先检查该 ULN2003 通道、P3 走线、插座接触和对应绕组。
- 四路都接近零：优先检查 3.3V 输入驱动兼容性、公共线和驱动板供电路径。
- 四路都相近且能获得明显带载压差：相序和逐相电气驱动基本排除，下一步检查电机批次、内部齿轮机构或更换已知正常的 5V 28BYJ-48/四输入 ULN2003 板交叉验证。

## 8. 验证与部署

实现前先添加主机单元测试，覆盖参数边界、单轮单通道掩码、3000ms 自动释放、显式中止、与运动互斥及计时回绕。协议生成物、Python/C 黄金布局、STM32 应用门禁和 ESP32 等待结果路径均需测试。

随后运行完整 Python 和 C 测试、STM32 交叉编译、Thumb 指令检查以及 Flash/RAM 预算检查。刷写 STM32 前保持电机 5V 断开，并提前告知操作员刷写会复位 MCU 和 OLED。部署后先验证无动作启动和正式链路恢复，再进入逐通道真机测量。
