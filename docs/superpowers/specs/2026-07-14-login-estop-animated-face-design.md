# 登录反馈、ESTOP 恢复与动态表情设计

- 日期：2026-07-14
- 状态：已确认
- 关联设计：`docs/superpowers/specs/2026-07-14-desktop-robot-design.md`

## 1. 背景

当前管理密码验证在 ESP32-S3 上需要数秒。网页提交密码后没有禁用状态、进度文字或阶段提示，并且在设备配置和状态加载完成前就隐藏登录层。手机端因此看起来像页面刷新，用户无法判断请求仍在处理、密码错误还是已经登录。

OLED 表情工具链已经通过真机验证：模型选择 `robot_set_expression`，ESP32 向 STM32 下发命令，STM32 返回 ACK。用户看到两个 X 且普通表情不变化，是因为 STM32 已进入 ESTOP；ESTOP 显示按安全优先级覆盖普通表情。当前 EC11 没有按压开关，固件又只允许本地按键确认解除 ESTOP，因此 ESP 重启或调试导致链路超时后只能复位 STM32。

恢复 IDLE 后，AI 的 happy 表情能够显示，但当前表情仅改变嘴形，画面静态且差异不明显。用户选择“情绪不随机、情绪内细节随机”：AI 决定情绪，STM32 在该情绪内生成眨眼、视线和嘴形变化。

## 2. 范围

### 2.1 目标

- 登录全过程提供明确、不可重复提交的阶段反馈。
- 密码错误、网络失败和设备数据加载失败使用不同提示。
- 为无按键版本提供经过网页二次确认的远程 ESTOP 恢复路径。
- STM32 在解除 ESTOP 前再次验证链路健康、运动已停止和当前状态。
- 普通表情由 STM32 本地程序化动画增强，不依赖持续网络请求。
- AI 情绪保持稳定，随机变化只影响同一情绪内的细节。
- ESTOP 和 FAULT 始终覆盖普通表情且不参与随机动画。

### 2.2 非目标

- 不允许 AI 自行解除 ESTOP。
- 不允许网页解除 FAULT。
- 不让普通表情随机切换情绪。
- 不从 ESP32 持续发送动画帧。
- 不增加位图表情资源或占用 W25Q 存储。
- 不改变电机硬限制或开放新的运动能力。

## 3. 登录状态反馈

### 3.1 状态模型

登录层使用以下显式状态：

```text
READY -> VERIFYING -> LOADING -> AUTHENTICATED
  ^          |           |
  +----------+-----------+
       密码、网络或加载错误
```

- `READY`：密码框和提交按钮可用。
- `VERIFYING`：提交后立即禁用密码框和按钮，按钮显示旋转状态及“正在验证”。
- `LOADING`：密码通过后清空密码值，继续保持登录层，显示“正在加载设备状态”。
- `AUTHENTICATED`：配置和状态均成功加载后才隐藏登录层。

首次设置管理密码时使用相同状态机，但 VERIFYING 文案为“正在初始化”。

### 3.2 错误反馈

- HTTP 401：显示“密码不正确”，恢复输入并聚焦、选中密码框。
- HTTP 409：显示“认证正在处理中”，不发起第二个请求。
- 网络、DNS 或连接中断：显示“连接失败，请检查网络后重试”。
- 登录成功但配置/状态加载失败：显示“登录成功，但设备状态加载失败”，提交按钮变为“重新连接”；重试时复用已有会话，不再次计算密码。
- 登录或加载超过 20 秒：终止本次浏览器等待，恢复可操作状态并显示超时提示。

状态文字放在 `aria-live="polite"` 区域。表单始终通过 JavaScript 拦截提交，不进行页面导航。按钮布局保持固定尺寸，旋转图标不会导致文字或表单跳动。

## 4. 远程解除 ESTOP

### 4.1 协议和设备链路

协议新增 `CLEAR_ESTOP` 命令，载荷只包含 `command_id`，继续使用现有 ACK/NACK 和去重机制：

```text
网页确认
  -> POST /api/v1/estop/clear
  -> DeviceService.clear_estop()
  -> MSG_CLEAR_ESTOP(command_id)
  -> STM32 安全校验
  -> ACK + IDLE / NACK + 错误码
```

STM32 仅在以下条件全部成立时 ACK：

- 当前状态为 ESTOP，而不是 FAULT。
- 当前 ESP 会话链路健康。
- 电机运动服务不处于 active 状态。
- 命令来自当前有效 mailbox 会话并通过现有帧校验。
- 用户已在受认证网页完成二次确认，该确认由独立 API 和命令表达。

成功后状态回到 IDLE、故障原因清零、选中模式设为 IDLE，并把普通表情重置为 neutral。失败不改变状态。现有未来物理按键解除路径继续保留。

核心状态函数现有的 `local_confirm` 参数重命名为 `operator_confirm`，表达“已由独立的人类操作确认”。物理按键短按和受认证网页命令可以分别提供该确认，AI 工具层不能调用此状态函数或构造 `CLEAR_ESTOP`。

### 4.2 网页入口

顶部红色急停按钮始终保留，不被恢复入口替换。设备在线且状态为 ESTOP 时，在“当前状态”面板底部显示“解除急停”按钮；其他状态隐藏该按钮。

点击后打开确认对话框：

- 标题：`解除急停？`
- 说明：`请确认机器人周围安全且车轮没有受阻。解除后机器人仅回到待机，不会自动恢复动作。`
- 默认焦点：`取消`
- 操作：`取消`、`确认解除`

确认请求期间禁用按钮。成功后刷新状态并显示 IDLE；失败时保留 ESTOP 和对话框，显示具体错误。接口要求登录、CSRF 和现有同源 Cookie。

### 4.3 表情命令与安全状态

STM32 在 ESTOP 或 FAULT 时拒绝普通 `SET_EXPRESSION`，避免命令 ACK 但被安全画面覆盖。ESP32 将单个工具的设备错误转换为结构化 `{ok: false, error: ...}` 工具结果并回传模型，使模型明确说明表情未改变，而不是声称成功。

## 5. STM32 程序化表情

### 5.1 模块边界

新增独立的 `face_animator` 模块，负责“基础情绪 -> 当前姿态”；`ui_service` 只负责把姿态绘制到 SSD1306 framebuffer 并分页刷新。

```text
AI/ESP SET_EXPRESSION
  -> base_expression
  -> face_animator_tick(now)
  -> face_pose {blink, gaze, mouth_variant}
  -> ui_service dirty
  -> SSD1306 分页刷新
```

动画器使用轻量 xorshift32 伪随机数，种子由 STM32 boot ID 和启动 tick 混合生成，不依赖额外硬件。单元测试可传固定种子获得确定结果。

### 5.2 动画规则

- 基础情绪保持到下一条 `SET_EXPRESSION`。
- 眨眼随机间隔 2–6 秒，闭眼保持约 160–240 ms。
- 视线在左、中、右三个位置间随机变化，间隔 3–7 秒。
- 嘴形在当前情绪允许的 2–3 个变体间变化，间隔 2.5–5 秒。
- neutral 使用轻微视线和眨眼，不自动变成其他情绪。
- happy 使用弯眼/普通眼变体和更明显的笑嘴。
- sad 使用下垂眼形和不同程度的下弯嘴。
- thinking 使用不对称眼睛、侧向视线和点状嘴，补齐当前缺失画法。
- surprised 使用高眼和 O 形嘴。
- sleepy 使用线状眼和小幅呼吸式嘴形。
- ESTOP、FAULT 使用固定安全画面，暂停随机动画。

状态文字 `IDLE`、`MAN`、`AI` 和链路指示点继续显示。姿态坐标受 128x64 边界限制，不覆盖底部状态文字。

### 5.3 刷新预算

动画器每次主循环都可 tick，但只有姿态真正变化时才设置 `dirty`。现有 OLED 分页刷新和页面间 SPI 服务策略保持不变；不为连续帧动画设置固定高帧率。任意动画事件之间至少留出完整分页刷新时间，避免增加 SPI `rx_errors`。

## 6. 测试

### 6.1 登录与网页

- 模拟延迟登录，断言按钮立即禁用并显示 VERIFYING。
- 登录成功后在配置和状态返回前保持登录层可见。
- 分别验证 401、409、网络错误、加载错误和超时文案。
- 断言表单提交不产生页面导航或重复登录请求。
- 在手机和桌面视口检查固定布局、旋转状态和长错误文字。
- 验证 ESTOP 时恢复按钮出现，其他状态隐藏。
- 验证确认、取消、ESC、遮罩、成功和失败路径。

### 6.2 协议与 STM32

- 生成并验证 `CLEAR_ESTOP` 的 Python/C ID 和载荷长度。
- 验证非 ESTOP、链路不健康、运动 active 和 FAULT 均拒绝解除。
- 验证合法解除回到 IDLE、清除故障原因且不启动电机。
- 验证重复 command ID 不重复改变状态。
- 验证 ESTOP/FAULT 下普通表情命令返回 BAD_STATE。

### 6.3 动态表情

- 固定种子下动画序列确定。
- 眨眼、视线和嘴形时间范围符合约束。
- 基础情绪不会被随机逻辑改变。
- ESTOP/FAULT 姿态固定并覆盖普通情绪。
- 无姿态变化时 OLED 不标脏、不刷新。
- 各姿态像素保持在 128x64 边界内。

### 6.4 真机

1. 部署后先启动 ESP32 正式应用，再复位 STM32，避免调试任务结束留下链路丢失 ESTOP。
2. 验证延迟登录期间手机显示明确进度，错误密码不关闭登录层。
3. 触发急停，使用网页二次确认恢复到 IDLE，确认电机不动作。
4. 依次下发 happy、sad、thinking、surprised 和 sleepy，观察基础情绪及随机细节。
5. 连续运行至少 60 秒，确认眨眼/视线自然、故障码为 0、SPI `rx_errors` 不增加。
6. 真机验收不发送 `robot_move`。

## 7. 完成标准

- 用户能明确区分登录处理中、密码错误、加载中和连接失败。
- 链路丢失 ESTOP 无需复位 STM32即可通过网页安全恢复。
- AI 在安全状态覆盖时不会错误声称普通表情已显示。
- 每个基础情绪具有明显差异并在情绪内自然变化。
- 全部 Python、C、协议和浏览器测试通过。
- 60 秒真机动画期间电机不动作，STM32 fault 为 0，SPI `rx_errors` 保持 0。
