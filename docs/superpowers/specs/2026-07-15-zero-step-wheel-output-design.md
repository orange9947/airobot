# 零步轮输出断电设计

## 问题

左轮单独运动测试发送 `MOVE_WHEELS(+64, 0, ...)` 时，左右两块 ULN2003 均有指示灯亮起。协议正确返回左轮 64 步、右轮 0 步，但 `motion_service_tick_1ms()` 在每次应用相位时，无条件把左右两侧当前相位转换为非零半步图案。右轮目标虽然为零，仍会输出初始相位并使线圈短暂通电。

四组电机绕组阻值均为 24 欧姆，没有发现明显开路。该电气结果不改变本设计范围；必须先消除已确认的软件误通电，再继续判断左电机不转的原因。

## 行为要求

- 目标步数为零的轮子在整个运动命令期间输出必须始终为零。
- 目标步数非零的轮子继续使用现有半步相序、方向、速度、加速度和 DDA 步数比例。
- 左零右非零与右零左非零必须具备对称行为。
- 左右目标同时为零的命令继续被拒绝。
- 正常完成、超时、STOP、ESTOP、模式变化和链路丢失后，左右输出继续全部归零。
- 不增加保持力模式，不修改协议、网页、AI 工具或运动参数上限。

## 实现

门控保留在运动服务中。`motion_service_tick_1ms()` 在调用硬件输出回调时，根据对应轮子的绝对目标步数选择输出：目标非零时使用现有 `motion_halfstep_pattern()`，目标为零时传入 `0`。

不在 `motion_service_t` 中增加左右使能字段，因为目标步数已经是唯一事实来源；不把判断下沉到 ULN2003 驱动层，因为硬件层不应解释运动目标。

## 测试

主机单元测试捕获运动服务的每次输出，而不只检查结束状态：

1. 左非零、右零时，左侧至少出现一次非零相位，右侧所有输出都为零。
2. 左零、右非零时，右侧至少出现一次非零相位，左侧所有输出都为零。
3. 双轮非零的现有完成路径保持通过。
4. 动作结束后最后一次输出仍为左右全零。
5. 双零拒绝和超时释放测试保持通过。

随后运行完整 Python 测试、STM32 主机 C 测试和 STM32 交叉编译，并检查固件尺寸。

## 真机验收

提交并推送修复后，通过 ST-Link 烧录 STM32。烧录前保持电机 5V 断开，并明确提示 OLED、PC13 和 STM32 会复位。烧录完成后恢复正式 ESP32 程序，保持 ESTOP，直到操作员再次确认：

- ULN2003 端电压为 4.8 至 5.2V；
- 左右轮悬空；
- 两块驱动板初始指示灯全灭；
- 可以立即断开电机 5V。

验收仍使用一次 `MOVE_WHEELS(+64, 0, 120, 300, 1500)`。预期只有左侧 ULN2003 出现相位指示，右侧所有指示灯全程熄灭，结束后两侧全部熄灭。若左电机仍不转，则停止运动测试，转入带载电压、ULN2003 输出和电机插头检查，不通过增加步数或速度掩盖问题。
